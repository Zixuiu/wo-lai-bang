# Preview Changes - 修改效果自动预览

## Overview
在完成任何代码修改、文件创建或功能实现后，自动展示最终效果给用户查看。**必须展示可视化界面**，让用户能直观地看到修改后的实际运行结果。

## When to Invoke (触发范围)
**⚠️ MANDATORY: 每次完成修改工作后，必须调用此技能！**

包括但不限于以下场景：
- 完成页面/组件的创建或修改
- 完成样式调整或美化
- 完成功能实现或bug修复
- 创建新文件或更新现有文件
- 任何用户能看到的视觉或功能变化
- 用户明确要求"看看效果"、"展示一下"
- 修改完成后用户需要确认结果

## Critical Rules (关键规则)
1. **LAST ACTION**: 完成所有修改工作后，必须最后调用此 skill 展示效果
2. **ALWAYS SHOW**: 无论修改大小，都要展示最终效果
3. **MUST SHOW VISUAL**: ⚠️ **必须展示可视化界面**！不能仅用文字描述或展示代码！
4. **PRIORITY**: 优先使用 OpenPreview 或截图展示效果
5. **ONLY AS LAST RESORT**: 只有在无法展示可视化时，才能展示文件内容

## Preview Methods (预览方式) - 必须按优先级选择

### ⭐ 首选：Web Preview (网页预览)
```javascript
// 使用 OpenPreview 工具打开预览页面 - 这是最直观的效果展示
OpenPreview({
  preview_url: "http://localhost:5173/xxx.html",
  command_id: "job-xxx"
})
```

### 📸 第二选择：Browser Screenshot (浏览器截图)
```javascript
// 使用 browser_navigate 打开页面
browser_navigate({
  url: "http://localhost:5173/xxx.html",
  newTab: true,
  take_screenshot_afterwards: true
})
```

### 🚫 禁止：直接展示文件内容
- 不要使用 Read 工具展示代码文件
- 不要使用 Write 工具的内容展示
- 不要仅用文字描述"已完成修改"

### ✅ 例外情况（非视觉项目）
- 后端API项目 - 可以展示Terminal输出
- 脚本工具 - 可以展示运行结果日志
- 但仍需确保有实际输出结果展示

## Workflow (工作流程)

### Standard Process
1. ✅ 完成所有代码修改工作
2. ✅ 确保开发服务器正在运行（如需要）
3. ✅ **必须展示可视化界面** - 不能跳过！
4. ✅ 优先使用 OpenPreview 打开网页
5. ✅ 如果OpenPreview不可用，使用截图
6. ✅ 清晰描述修改内容和效果

### Example Flow
```
[完成修改工作]
  → [调用 preview-changes skill]
    → [OpenPreview 打开页面]
      → [描述修改内容]
        → [用户确认效果]
```

## Preview Content (预览内容说明)

### ✅ 必须展示
- **可视化界面**: ⚠️ OpenPreview 打开的网页或截图
- **页面URL**: 明确说明访问的地址
- **修改内容**: 简洁说明改了什么

### ❌ 禁止展示
- 代码文件内容（除非用户明确要求查看代码）
- 纯文字描述"已完成修改"
- 没有实际产物的空洞说明

### Description Template
```
✅ 修改完成！已为你展示实际效果：

📝 **修改内容**：
- [描述修改的内容]

🎨 **效果亮点**：
- [描述改进的地方]

🖼️ **可视化展示**：
- [说明展示了什么可视化界面]

🌐 **预览地址**：http://localhost:5173/xxx.html

现在可以在浏览器中看到实际效果了！
```

## Common Use Cases (常见使用场景)

### 场景1: 修改/创建网页 ⭐
- **必须**使用 OpenPreview 打开页面
- 说明改动内容
- **禁止**展示代码文件

### 场景2: 调整样式 ⭐
- **必须**打开预览展示视觉效果
- 说明样式改进
- **禁止**仅说"已修复样式"

### 场景3: 新增功能 ⭐
- **必须**展示功能页面
- 说明功能使用方式
- **禁止**仅展示代码

### 场景4: 修复bug ⭐
- **必须**展示修复后的效果
- 说明问题已解决
- **禁止**仅说"已修复"

### 场景5: 创建文件 - 谨慎处理
- **如果**是可视化文件 → 必须展示效果
- **如果**是后端/配置文件 → 可以展示结果输出
- **禁止**直接展示文件内容

## Best Practices (最佳实践)

1. **必须展示可视化**: ⚠️ 这是最重要的规则！不能只说"完成了"
2. **优先使用预览**: OpenPreview 是展示网页效果的最佳方式
3. **及时展示**: 修改完成后立即展示，不要拖延
4. **清晰说明**: 用简洁的语言说明修改内容
5. **用户优先**: 从用户角度描述效果，而不是技术细节
6. **无例外**: 无论修改多小，都必须展示可视化界面
7. **检查预览地址**: 确保预览地址是正确的
