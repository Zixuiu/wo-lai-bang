# Preview Changes - 修改效果自动预览

## Overview
在完成任何代码修改、文件创建或功能实现后，自动展示最终效果给用户查看。确保用户能直观地看到修改后的实际运行结果。

## When to Invoke (触发范围)
**⚠️ MANDATORY: 每次完成修改工作后，必须调用此技能！**

包括但不限于以下场景：
- 完成页面/组件的创建或修改
- 完成样式调整或美化
- 完成功能实现或bug修复
- 创建新文件或更新现有文件
- 任何用户能看到的视觉或功能变化
- 用户明确要求"看看效果"、"展示一下"
- 修改完成后用户需要确认结果

## Critical Rules (关键规则)
1. **LAST ACTION**: 完成所有修改工作后，必须最后调用此 skill 展示效果
2. **ALWAYS SHOW**: 无论修改大小，都要展示最终效果
3. **USE PREVIEW**: 优先使用 OpenPreview 工具打开预览页面
4. **SCREENSHOT**: 如果有浏览器可用，尽量截图展示效果
5. **DESCRIBE**: 清晰描述修改了什么、展示的是什么

## Preview Methods (预览方式)

### 1. Web Preview (网页预览) - 首选
```javascript
// 使用 OpenPreview 工具打开预览页面
OpenPreview({
  preview_url: "http://localhost:5173/xxx.html",
  command_id: "job-xxx"
})
```

### 2. Browser Screenshot (浏览器截图) - 如果可用
- 使用 browser_navigate 打开页面
- 使用 browser_take_screenshot 截图
- 展示截图给用户

### 3. File Display (文件展示) - 非网页项目
- 读取关键文件内容展示
- 展示文件结构变化
- 使用 cat 或 Read 工具展示

### 4. Terminal Output (终端输出) - 后端/脚本项目
- 展示运行结果日志
- 展示测试输出
- 展示编译/构建结果

## Workflow (工作流程)

### Standard Process
1. ✅ 完成所有代码修改工作
2. ✅ 确保开发服务器正在运行（如需要）
3. ✅ 调用此技能展示效果
4. ✅ 使用最合适的预览方式
5. ✅ 清晰描述修改内容和效果

### Example Flow
```
[完成修改工作]
  → [调用 preview-changes skill]
    → [OpenPreview 打开页面]
      → [描述修改内容]
        → [用户确认效果]
```

## Preview Content (预览内容说明)

### What to Show
- **页面URL**: 明确说明访问的地址
- **修改内容**: 简洁说明改了什么
- **效果亮点**: 指出改进或新增的功能
- **技术细节**: 可选，重要的实现细节

### Description Template
```
✅ 修改完成！已为你打开预览页面：

📝 **修改内容**：
- [描述修改的内容]

🎨 **效果亮点**：
- [描述改进的地方]

🌐 **预览地址**：http://localhost:5173/xxx.html

现在可以在浏览器中看到效果了！
```

## Common Use Cases (常见使用场景)

### 场景1: 修改/创建网页
- 使用 OpenPreview 打开页面
- 说明改动内容

### 场景2: 调整样式
- 打开预览展示视觉效果
- 说明样式改进

### 场景3: 新增功能
- 展示功能页面
- 说明功能使用方式

### 场景4: 修复bug
- 展示修复后的效果
- 说明问题已解决

## Best Practices (最佳实践)

1. **及时展示**: 修改完成后立即展示，不要拖延
2. **清晰说明**: 用简洁的语言说明修改内容
3. **确保可访问**: 确保预览地址是可访问的
4. **多方式结合**: 预览 + 描述 + 截图（如果可能）
5. **用户优先**: 从用户角度描述效果，而不是技术细节
